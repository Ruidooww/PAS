# M15 客情画像 8 维定义与采集边界

## 1. 文档定位

本文档是 M15 客情画像 Phase B 画像构建前置业务输入，用于定义客户画像 8 个业务维度、字段来源、合规边界和后续工程字段表准入原则。

本文档不做工程实现，不修改 API、数据库 schema、前端页面、画像算法、评分模型、自动推荐、RAGFlow、FastGPT、OpenAI、Agent、Workflow、PluginManager、多租户或平台化能力。

## 3. 合规与使用原则

1. 只采集与客户画像、客情判断、销售推进、项目风险识别直接相关的业务信息。
2. 不采集与业务无关的个人隐私，包括家庭、健康、个人财务、身份证件、政治宗教、个人社交关系、私人行程等信息。
3. 不基于个人隐私、敏感身份或非业务行为进行客户价值、成交概率、风险等级判断。
4. 涉及联系人姓名、电话、邮箱、职位等个人信息时，应控制字段范围、访问权限、展示脱敏和审计记录。
5. 系统输出应以“建议”“风险提示”“待确认事项”为主，不输出“必成交”“必失败”“客户一定会采购”等确定性结论。
6. 自动分析只能基于已授权、可追溯、可解释的业务资料，不得从无授权邮件、私人聊天、个人社交媒体或非业务材料中抽取信息。
7. 所有客户敏感、监管敏感字段进入工程字段表前，需要法务 / 合规确认采集、展示、分析和沉淀规则。

## 4. 合规分级

| 分级 | 含义 | 处理要求 |
|---|---|---|
| 公开 | 客户官网、公开招投标公告、公开新闻、公开工商信息等可公开获取的信息 | 可采集、展示和分析，仍需保留来源 |
| 内部 | PAS 内部售前、销售、交付过程中形成的普通业务信息 | 限内部授权角色访问，保留来源和更新时间 |
| 客户敏感 | 预算、采购计划、技术环境、联系人、组织关系、竞品态势、业务痛点等可能影响客户利益的信息 | 最小权限访问，展示脱敏或按角色控制，进入字段表前需合规确认 |
| 监管敏感 | 涉及法律合规要求、招投标规则、监管审计、客户安全事件或安全缺陷的信息 | 严格权限、审计追踪、合规确认后方可采集和分析 |

## 5. 来源边界

| 来源 | 是否可作为候选来源 | 边界 |
|---|---|---|
| CRM 字段 | 是 | 仅限客户、联系人、商机、跟进、阶段、金额、任务等业务字段 |
| 售前手填 | 是 | 需保留填写人、填写时间和依据，避免主观臆测 |
| 会议纪要 | 是 | 仅使用已授权业务会议纪要；涉及个人隐私的内容不得入库 |
| 邮件 / IM 摘要 | 条件允许 | 仅使用企业授权账号、业务往来内容和脱敏摘要；不采集私人聊天或无关内容 |
| 招投标文件 | 是 | 只采集项目要求、采购流程、评分规则、技术条款和公开披露信息 |
| 方案交互记录 | 是 | 只采集客户对方案、报价、功能、部署方式的业务反馈 |
| 实施 / 售后记录 | 条件允许 | 仅用于存量客户画像和风险提示，需遵守客户合同和数据隔离要求 |
| 客户公开资料 | 是 | 需记录来源，不将公开资料外推为未经证实的内部事实 |
| 个人社交媒体 / 私人关系 | 否 | 不采集、不展示、不分析 |

## 7. 维度 1：客户基础信息

### 业务定义

客户基础信息用于识别客户主体、行业、规模、区域、组织类型、存量关系和基础分层，是客户画像、商机归属、方案行业化和售前资源分配的基础。

### 为什么需要

该维度帮助售前判断客户所处行业、组织规模、区域交付条件和历史合作基础，避免在后续画像中混淆客户主体、集团子公司、存量客户和新客户。

### 建议字段

| 字段 | 取值类型 | 字段来源 | 敏感 / 合规风险 | 允许系统自动分析 | 可作为建议 / 风险 / 优先级依据 | 工程字段表准入 |
|---|---|---|---|---|---|---|
| 客户名称 | 文本 | CRM、客户公开资料、招投标文件、售前手填 | 公开 / 内部 | 是 | 是 | 可进入 |
| 所属行业 | 枚举 / 多标签 | CRM、公开资料、售前手填 | 公开 / 内部 | 是 | 是 | 可进入 |
| 所在区域 | 枚举 | CRM、公开资料、售前手填 | 公开 / 内部 | 是 | 是 | 可进入 |
| 企业规模 | 枚举 / 数值区间 | CRM、公开资料、售前手填 | 公开 / 内部 | 是 | 是 | 可进入 |
| 客户类型 | 枚举 | CRM、销售手填 | 内部 | 是 | 是 | 可进入 |
| 是否集团型客户 | 布尔 | CRM、公开资料、售前手填 | 内部 | 是 | 是 | 可进入 |
| 是否存量客户 | 布尔 | CRM、历史商机、实施 / 售后记录 | 内部 | 是 | 是 | 可进入 |

### 示例

某华东制造业集团为存量客户，存在多个子公司和跨区域终端，后续方案需按集团总部决策与子公司落地分开判断。

### 不应采集或推断

不得采集客户经营者或员工的家庭背景、私人资产、个人社会关系；不得仅凭行业或区域推断客户一定有某类安全事件、采购倾向或预算能力。

## 8. 维度 2：组织与关键人关系

### 业务定义

组织与关键人关系用于描述客户内部相关部门、关键业务联系人、技术影响人、采购影响人、使用部门和高层关注人，帮助判断沟通路径和决策链条。

### 为什么需要

IP-Guard 类项目通常涉及信息安全、IT、业务部门、采购、法务和管理层。明确业务角色和影响关系，可以减少推进盲区，识别决策链不清、关键人缺失、需求传递失真等风险。

### 建议字段

| 字段 | 取值类型 | 字段来源 | 敏感 / 合规风险 | 允许系统自动分析 | 可作为建议 / 风险 / 优先级依据 | 工程字段表准入 |
|---|---|---|---|---|---|---|
| 关键联系人 | 结构化对象 | CRM、售前手填、会议纪要 | 客户敏感，涉及个人信息 | 条件允许 | 是，但需角色权限 | 合规确认后进入 |
| 部门 | 文本 / 枚举 | CRM、会议纪要、售前手填 | 客户敏感 | 是 | 是 | 合规确认后进入 |
| 职位 / 角色 | 文本 / 枚举 | CRM、会议纪要、售前手填 | 客户敏感，涉及个人信息 | 是 | 是 | 合规确认后进入 |
| 决策影响力 | 枚举 | 售前手填、会议纪要 | 客户敏感，主观判断 | 条件允许，需标注置信度 | 是，作为待确认事项 | 合规确认后进入 |
| 技术影响人 | 结构化对象 | CRM、售前手填、会议纪要 | 客户敏感，涉及个人信息 | 条件允许 | 是 | 合规确认后进入 |
| 采购影响人 | 结构化对象 | CRM、售前手填、会议纪要 | 客户敏感，涉及个人信息 | 条件允许 | 是 | 合规确认后进入 |
| 使用部门 | 文本 / 多标签 | 会议纪要、售前调研、方案交互 | 客户敏感 | 是 | 是 | 合规确认后进入 |
| 高层关注人 | 结构化对象 | CRM、售前手填、会议纪要 | 客户敏感，涉及个人信息 | 条件允许 | 是，需谨慎展示 | 合规确认后进入 |

### 示例

信息安全部门提出数据防泄漏需求，IT 部门负责技术验证，采购部门负责流程，高层关注合规检查结果。系统可以提示“决策链仍需确认采购与高层关注人”，不能断言某个人会决定采购。

### 不应采集或推断

不得采集私人手机号以外的非业务联系方式、家庭情况、个人偏好、私人关系、情绪倾向、健康或财务信息；不得推断个人立场、忠诚度、可被影响程度或非业务动机。

## 9. 维度 3：业务场景与安全需求

### 业务定义

业务场景与安全需求用于描述客户当前业务痛点、信息安全目标、应用场景和合规要求，是方案推荐、产品能力匹配和售前材料准备的核心依据。

### 为什么需要

客户是否关注数据防泄漏、终端管控、文件加密、防勒索、外发管控、审计追溯、合规检查或国产化，会直接决定方案重点、演示内容、测试范围和竞品对比口径。

### 建议字段

| 字段 | 取值类型 | 字段来源 | 敏感 / 合规风险 | 允许系统自动分析 | 可作为建议 / 风险 / 优先级依据 | 工程字段表准入 |
|---|---|---|---|---|---|---|
| 数据防泄漏需求 | 多标签 / 优先级 | 需求调研、会议纪要、方案交互 | 客户敏感 | 是 | 是 | 可进入 |
| 终端管控需求 | 多标签 / 优先级 | 需求调研、会议纪要、方案交互 | 客户敏感 | 是 | 是 | 可进入 |
| 文件加密需求 | 多标签 / 优先级 | 需求调研、方案交互、招投标文件 | 客户敏感 | 是 | 是 | 可进入 |
| 防勒索需求 | 多标签 / 优先级 | 需求调研、会议纪要 | 客户敏感 / 监管敏感 | 条件允许 | 是，作为风险提示 | 合规确认后进入 |
| 外发管控需求 | 多标签 / 优先级 | 需求调研、方案交互、招投标文件 | 客户敏感 | 是 | 是 | 可进入 |
| 审计追溯需求 | 多标签 / 优先级 | 需求调研、招投标文件、合规要求 | 客户敏感 / 监管敏感 | 条件允许 | 是 | 合规确认后进入 |
| 合规检查需求 | 多标签 / 自由文本 | 招投标文件、会议纪要、售前手填 | 监管敏感 | 条件允许 | 是，需标注依据 | 合规确认后进入 |
| 国产化 / 信创要求 | 枚举 / 自由文本 | 招投标文件、需求调研、公开政策 | 内部 / 客户敏感 | 是 | 是 | 可进入 |

### 示例

客户明确提出外发审批、U 盘管控、研发资料加密和审计追溯需求。系统可以建议优先准备数据防泄漏与终端管控方案材料，并提示“审计要求需确认等保或行业合规条款”。

### 不应采集或推断

不得未经证实地记录客户发生过安全事故、违规事件或内部泄密；不得把“关注防勒索”推断为“已发生勒索攻击”；不得采集与项目无关的业务秘密。

## 10. 维度 4：当前系统与技术环境

### 业务定义

当前系统与技术环境用于描述客户已有 IT、安全、终端、网络、身份、数据流转和部署条件，是判断方案可落地性、测试复杂度和实施风险的依据。

### 为什么需要

IP-Guard 类项目部署效果受终端数量、服务器环境、AD / LDAP / IAM、已有安全产品、网络隔离、内外网环境和数据流转方式影响。缺少该维度会导致方案评估和交付风险判断失真。

### 建议字段

| 字段 | 取值类型 | 字段来源 | 敏感 / 合规风险 | 允许系统自动分析 | 可作为建议 / 风险 / 优先级依据 | 工程字段表准入 |
|---|---|---|---|---|---|---|
| 终端数量 | 数值区间 | 需求调研、CRM、售前手填 | 客户敏感 | 是 | 是 | 合规确认后进入 |
| 服务器环境 | 枚举 / 自由文本 | 需求调研、技术交流、招投标文件 | 客户敏感 | 是 | 是 | 合规确认后进入 |
| AD / LDAP / IAM 情况 | 枚举 / 自由文本 | 技术交流、需求调研 | 客户敏感 | 是 | 是 | 合规确认后进入 |
| 已有安全产品 | 多标签 | 技术交流、售前手填、招投标文件 | 客户敏感 | 是 | 是 | 合规确认后进入 |
| 现有厂商 | 多标签 / 文本 | 技术交流、招投标文件、售前手填 | 客户敏感 | 是 | 是，注意竞品合规表达 | 合规确认后进入 |
| 部署方式 | 枚举 | 技术交流、方案交互 | 客户敏感 | 是 | 是 | 合规确认后进入 |
| 内外网环境 | 枚举 / 自由文本 | 技术交流、需求调研 | 客户敏感 / 监管敏感 | 条件允许 | 是，作为实施风险提示 | 合规确认后进入 |
| 数据流转方式 | 自由文本 / 多标签 | 需求调研、业务访谈 | 客户敏感 / 监管敏感 | 条件允许 | 是，作为方案设计依据 | 合规确认后进入 |

### 示例

客户终端规模约 3000 台，存在办公网和生产网隔离，使用 AD 管理账号，已有终端安全产品。系统可以提示“需确认跨网部署方式和已有安全产品兼容性”。

### 不应采集或推断

不得采集服务器密码、密钥、漏洞细节、内网拓扑图原件、未授权日志、账号清单；不得推断客户存在安全缺陷或合规违规，除非有客户授权材料明确支持。

## 11. 维度 5：项目阶段与推进状态

### 业务定义

项目阶段与推进状态用于判断商机或项目当前处于初步接触、需求调研、方案交流、测试验证、商务报价、招投标、合同推进、实施交付或售后维护阶段。

### 为什么需要

不同阶段需要不同售前动作和材料。阶段判断可以帮助系统提示下一步待办、风险缺口和销售节奏，但不能替代销售或项目负责人的判断。

|---|---|---|---|---|---|---|
| 初步接触 | 阶段枚举 | CRM、销售手填 | 内部 | 是 | 是 | 可进入 |
| 需求调研 | 阶段枚举 | CRM、会议纪要、售前手填 | 内部 | 是 | 是 | 可进入 |
| 方案交流 | 阶段枚举 | CRM、方案交互记录 | 内部 | 是 | 是 | 可进入 |
| 测试验证 | 阶段枚举 | CRM、测试记录、售前手填 | 内部 / 客户敏感 | 是 | 是 | 可进入 |
| 商务报价 | 阶段枚举 | CRM、报价记录 | 客户敏感 | 条件允许 | 是 | 合规确认后进入 |

| 售后维护 | 阶段枚举 | 售后记录、CRM | 客户敏感 | 是 | 是 | 合规确认后进入 |

### 示例

项目已完成需求调研并进入测试验证，但尚未确认采购流程。系统可以提示“测试通过不等于商务推进成立，需补充预算与采购机制确认”。

### 不应采集或推断

不得把客户一次口头反馈直接推断为正式阶段变更；不得自动将“测试中”推断为“高成交概率”；阶段变更应保留来源和人工确认记录。

## 12. 维度 6：预算、采购与决策机制

### 业务定义

预算、采购与决策机制用于判断项目是否具备真实采购条件，包括预算、预算周期、采购方式、决策流程、招投标要求、竞品或指定技术路线、预计采购时间等。

### 为什么需要

该维度帮助识别项目是否有明确预算和采购路径，避免售前资源投入到无法推进或采购机制不清的项目；同时防止销售动作触碰招投标和采购合规红线。

### 建议字段

| 字段 | 取值类型 | 字段来源 | 敏感 / 合规风险 | 允许系统自动分析 | 可作为建议 / 风险 / 优先级依据 | 工程字段表准入 |
|---|---|---|---|---|---|---|
| 是否有预算 | 枚举 | CRM、销售手填、会议纪要 | 客户敏感 | 条件允许 | 是，需标注依据 | 合规确认后进入 |
| 预算周期 | 枚举 / 日期区间 | CRM、销售手填、会议纪要 | 客户敏感 | 条件允许 | 是 | 合规确认后进入 |
| 采购方式 | 枚举 | CRM、招投标文件、销售手填 | 客户敏感 / 监管敏感 | 条件允许 | 是，作为合规提示 | 合规确认后进入 |
| 决策流程 | 自由文本 / 流程枚举 | 会议纪要、销售手填 | 客户敏感 | 条件允许 | 是，作为待确认事项 | 合规确认后进入 |
| 是否需要招标 | 布尔 / 枚举 | 招投标文件、CRM、销售手填 | 监管敏感 | 条件允许 | 是，作为合规提示 | 合规确认后进入 |
| 是否已有竞品 | 布尔 / 多标签 | 会议纪要、招投标文件、销售手填 | 客户敏感 | 是 | 是 | 合规确认后进入 |
| 是否存在指定品牌或技术路线 | 布尔 / 自由文本 | 招投标文件、会议纪要、销售手填 | 监管敏感 | 条件允许 | 是，作为合规风险提示 | 合规确认后进入 |
| 预计采购时间 | 日期区间 | CRM、销售手填、会议纪要 | 客户敏感 | 条件允许 | 是 | 合规确认后进入 |

### 示例

客户表示年度预算已预留，但采购需走公开招标，预计在第三季度启动。系统可以提示“准备招投标技术材料，避免输出规避采购流程的建议”。

### 不应采集或推断

不得引导虚假承诺、规避招标、拆分采购、绕过客户采购规则；不得记录或推断客户内部非公开审批意见、个人利益安排或不当关系。

## 13. 维度 7：风险、阻力与竞争态势

### 业务定义

风险、阻力与竞争态势用于识别项目推进中的竞品压力、客户异议、技术风险、商务风险、合规风险、预算风险、决策链不清和延期风险。

### 为什么需要

该维度帮助售前和销售提前识别项目风险，制定验证、澄清、材料补齐和管理层沟通计划，避免只看商机金额而忽略真实阻力。

### 建议字段

| 字段 | 取值类型 | 字段来源 | 敏感 / 合规风险 | 允许系统自动分析 | 可作为建议 / 风险 / 优先级依据 | 工程字段表准入 |
|---|---|---|---|---|---|---|
| 竞品厂商 | 多标签 / 文本 | CRM、会议纪要、招投标文件、销售手填 | 客户敏感 | 是 | 是 | 合规确认后进入 |
| 客户异议 | 多标签 / 自由文本 | 会议纪要、方案交互、销售手填 | 客户敏感 | 是 | 是 | 合规确认后进入 |
| 技术风险 | 多标签 / 风险等级 | 技术交流、测试记录、售前手填 | 客户敏感 | 是 | 是 | 合规确认后进入 |
| 商务风险 | 多标签 / 风险等级 | CRM、销售手填、商务记录 | 客户敏感 | 条件允许 | 是 | 合规确认后进入 |
| 合规风险 | 多标签 / 风险等级 | 招投标文件、法务意见、销售手填 | 监管敏感 | 条件允许 | 是，需标注依据 | 合规确认后进入 |
| 预算风险 | 多标签 / 风险等级 | CRM、会议纪要、销售手填 | 客户敏感 | 条件允许 | 是 | 合规确认后进入 |
| 决策链不清风险 | 风险等级 | CRM、会议纪要、销售手填 | 客户敏感 | 条件允许 | 是，作为待确认事项 | 合规确认后进入 |
| 项目延期风险 | 风险等级 / 日期 | CRM、项目计划、销售手填 | 客户敏感 | 是 | 是 | 合规确认后进入 |

### 示例

客户正在对比竞品，技术侧关注终端性能影响，采购流程未确认。系统可以提示“当前主要风险为竞品压力、测试指标未闭环、采购机制不清”，并建议补齐测试报告和采购路径确认。

### 不应采集或推断

不得编造竞品关系、内部阻力或客户负面评价；不得记录不当商业竞争信息、个人关系线索或未经授权的竞品商业秘密；不得自动判定项目失败。

## 14. 维度 8：客户价值与跟进策略

### 业务定义

客户价值与跟进策略用于综合判断客户等级、商机价值、成交概率区间、预计金额、跟进优先级、下一步动作、推荐跟进人、推荐材料和风险提示。

### 为什么需要

该维度把前 7 个维度转化为销售和售前可执行的行动建议，帮助团队分配资源、准备材料、安排会议和识别风险。但最终销售动作必须由业务人员判断。

### 建议字段

| 字段 | 取值类型 | 字段来源 | 敏感 / 合规风险 | 允许系统自动分析 | 可作为建议 / 风险 / 优先级依据 | 工程字段表准入 |
|---|---|---|---|---|---|---|
| 客户等级 | 枚举 | CRM、销售管理规则、售前手填 | 内部 | 是 | 是 | 可进入 |
| 商机价值 | 金额区间 / 等级 | CRM、报价记录、销售手填 | 客户敏感 | 条件允许 | 是 | 合规确认后进入 |
| 成交概率 | 概率区间 / 等级 | CRM、销售手填、阶段和风险信号 | 客户敏感，主观判断 | 条件允许，必须保留解释 | 是，不能输出确定性结论 | 合规确认后进入 |
| 预计金额 | 金额区间 | CRM、报价记录、销售手填 | 客户敏感 | 条件允许 | 是 | 合规确认后进入 |
| 跟进优先级 | 枚举 | 系统建议、销售手填 | 内部 | 是 | 是 | 可进入 |
| 下一步动作 | 多标签 / 自由文本 | 系统建议、销售手填、售前手填 | 内部 | 是 | 是 | 可进入 |
| 推荐跟进人 | 结构化对象 | CRM、销售管理规则 | 内部 / 个人信息 | 条件允许 | 是，需角色权限 | 合规确认后进入 |
| 推荐材料 | 多标签 | 产品知识库、方案模板、售前手填 | 内部 | 是 | 是 | 可进入 |
| 风险提示 | 多标签 / 自由文本 | 维度 2-7 汇总、销售手填 | 客户敏感 | 是，需可解释 | 是，作为风险提示 | 合规确认后进入 |

### 示例

客户为集团型存量客户，需求明确但采购机制未确认。系统可以建议“高优先级跟进，下一步确认预算周期和采购方式，准备集团部署案例与测试清单”，同时提示“成交概率仅为区间建议，需要销售确认”。

### 不应采集或推断

不得输出“必成交”“必失败”“客户一定会采购”等确定性结论；不得基于联系人个人隐私、私人关系或非业务因素生成优先级；不得把系统建议作为自动销售决策。

## 15. 8 维依赖与冲突

| 关系 | 说明 | 处理原则 |
|---|---|---|
| 基础信息 → 其他维度 | 客户主体、集团关系、存量状态影响组织、需求、项目阶段和价值判断 | 客户主体不清时，不应合并画像 |
| 组织关系 → 项目阶段 / 采购机制 | 决策链和关键角色影响阶段真实性与采购路径判断 | 决策链不清只能提示风险，不能推断成交概率 |
| 业务需求 → 技术环境 | 需求需要结合现有系统和部署条件判断可行性 | 技术环境缺失时，方案建议应标注“待技术确认” |
| 项目阶段 → 客户价值 | 阶段越靠后通常行动更明确，但不能单独决定优先级 | 阶段必须结合预算、风险、客户等级判断 |
| 预算采购 → 风险态势 | 预算不明、采购方式不明、招标要求不明会形成推进风险 | 不得给出规避采购规则的建议 |
| 风险态势 → 跟进策略 | 风险提示会影响下一步动作、材料准备和优先级 | 风险必须有来源，不得凭空生成 |
| 客户价值 ← 前 7 维汇总 | 客户价值与跟进策略依赖前 7 维综合输入 | 只输出建议、区间和待确认事项 |

### 互斥或冲突示例

1. `是否有预算 = 否` 与 `预计采购时间 = 近期` 冲突时，应提示“采购时间依据不足”，不能提高成交概率。
2. `项目阶段 = 合同推进` 但 `决策流程 = 未确认` 时，应提示“阶段真实性待确认”。
3. `部署方式 = 内网部署` 但 `数据流转方式 = 跨公网同步` 时，应提示“技术边界冲突待确认”。
4. `是否需要招标 = 是` 但 `下一步动作 = 绕过招标直接签约` 属于不合规建议，系统不得生成。
5. `竞品厂商 = 未确认` 时，不得自动生成针对某竞品的负面判断。

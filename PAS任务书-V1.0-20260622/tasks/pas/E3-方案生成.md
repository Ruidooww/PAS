## 1. 模块定位

PAS 区别于"通用问答工具"的核心。售前输入客户需求，系统检索知识库 + 按模板组装 + LLM 生成，产出 IP-Guard 方案初稿。**精简版聚焦"能出一份结构合理、内容有依据、可人工修订的初稿"**，不追求全自动确定性生成。

## 2. 用户故事

- 作为**售前**，我填客户行业/规模/痛点，系统自动生成一份 IP-Guard 部署方案初稿
- 作为**售前**，生成的方案每段有知识来源，我能快速核对、修订、定稿
- 作为**售前**，我能把定稿方案导出 Word 发给客户

## 3. 功能规格

### 3.1 需求结构化（E3.1）

```
输入：自由文本 + 表单字段
  - 客户名 / 行业 / 规模（终端数）
  - 核心诉求（多选：文档加密 / 行为审计 / 准入控制 ...）
  - 特殊约束（自由文本）

处理：LLM 辅助补全 → requirement_json
  { customer, industry, scale, needs[], constraints[] }
```

- 表单为主，自由文本经 LLM 抽取补全缺失字段
- 存 `Proposal.requirement_json`

### 3.2 方案模板体系（E3.2）

```
模板 = 章节骨架 + 每章节的检索意图 + 变量占位
IP-Guard 标准方案模板（首版 1 套）：
  1. 项目背景       ← 填充 customer/industry
  2. 需求分析       ← requirement_json
  3. 方案概述       ← 检索"IP-Guard 产品架构"
  4. 功能模块设计   ← 按 needs[] 逐项检索对应功能说明
  5. 部署架构       ← 检索"部署架构 + scale 适配"
  6. 实施计划       ← 模板固定 + 微调
  7. 服务与支持     ← 模板固定
```

- 模板存配置（`config/proposal-templates/`），不硬编码
- 每章节声明"检索意图"，驱动 E3.3 检索

### 3.3 生成流水线（E3.3）★ 异步 — 2026-06-23 修订为 PAS 自编排

```
PAS BullMQ worker（D5 异步化）：
  job: { proposalId, requirementJson, templateId, userId }
  steps:
    for 每个章节 in 模板:
      if 章节 fixed:
        1. 直接拼模板 + 变量（不检索）
      else:
        1. ragflowClient.retrieve({
             query: 章节 retrievalIntent + requirementJson 关键词,
             kbId: PAS_KB_ID,
             topK: 5,
             // 实际 RAGFlow MCP/REST 参数走 ragflowClient 默认值（见 #5）：
             //   rerank_id = gte-rerank-v2@bailian@Tongyi-Qianwen
             //   page_size = 30
             docIdWhitelist: AclService.computeVisibleDocIds(userId)
           })
        2. 拼章节 prompt：{ requirementJson, retrieved chunks, variables }
        3. llmClient.complete(prompt, { temperature, maxTokens 走 runtimeConfig })
        4. 解析 LLM 输出里的 [n] 引用 → 映射回 chunks，得 refs[]
      5. 收集 { title, body, refs[] }
    组装 → content_json（含每段 refs）
    写 Proposal.content_json
    SSE 推进度（第 N/7 章生成中）
```

- 每章独立检索 + 生成，互不污染上下文
- 失败章节可单独重试（job retry 3 次，配置见 #24 runtimeConfig）
- **PAS 自编排，不走 FastGPT workflow**——决策反转见 [ADR-001 § 决策修订记录](../../decisions/ADR-001-pas-fastgpt-ragflow.md) + `docs/experiments/exp-001-ragflow-mcp-fastgpt.md`
- 与 E2 同模式（PAS 直连 RAGFlow），统一架构、统一审计、统一测试基建
- ACL 过滤复用 E1.5 `AclService`，userId 直接从 BullMQ job 带入

### 3.4 方案 CRUD + 版本（E3.4）

- `Proposal` 增删改查
- 版本：每次定稿/重大修订 +1，保留历史
- 状态：`draft`（生成中/可编辑）→ `final`（定稿）

### 3.5 导出（E3.5）

- 首版：Markdown → Word（`docx` 库 / mammoth 反向）
- 含出处脚注
- **PPT 后置**：起步不做 PPT；真要时按任务书 §4 拆 Python(`python-pptx`) 或 `pptxgenjs`

### 3.6 方案前端（E3.6）

- 需求录入表单
- 生成进度（逐章节）
- 富文本预览 + 分段修订（每段可见出处）
- 导出按钮

## 4. 数据模型增量

```prisma
model Proposal {
  id              String   @id @default(cuid())
  customerRef     String   // 指向 CustomerMirror.ref（第三方客户 id；阶段二可迁本地）
  opportunityRef  String?
  title           String
  status          String   @default("draft")  // draft|final
  requirementJson Json
  contentJson     Json?    // { sections: [{ title, body, refs[] }] }
  pptFileKey      String?
  version         Int      @default(1)
  createdBy       String
  createdAt       DateTime @default(now())
  @@index([customerRef])
}
```

## 5. 关键 Prompt（`prompts/proposal_section.txt`）

要点：
- 基于检索内容写本章节，禁止编造产品能力
- 引用具体功能/配置时标出处
- 语气专业、面向客户（非内部技术口吻）
- 缺少检索依据的部分 → 标 `[待补充]`，不臆造

## 6. 验收标准（DoD）

- [ ] 填需求 → 异步生成 7 章节方案初稿
- [ ] 每段内容可见知识来源
- [ ] 方案可分段修订、存版本
- [ ] 导出 Word 成功
- [ ] 方案通过 `customer_ref` 关联到第三方 CRM 客户（E4 联动）
- [ ] 生成失败的章节可单独重试

## 7. 不在本模块（后置 v2）

- PPT 自动生成引擎
- 方案质量评估 + Outcome Learning + LoRA 微调
- 竞品对比章节自动生成

# E1 权限基座 — 详细 spec

> 配套 [PAS-任务清单-V1.0.md](../../PAS-任务清单-V1.0.md) ｜ 里程碑 M0-M1
> 核心原则：D3（权限在代码边界，不在 prompt）+ D4（API 层脱敏）

---

## 1. 模块定位

PAS 的安全地基。所有业务模块（问答/方案/客户）的访问控制都依赖它。外部访客（公众号/客服）与内部员工（飞书）走**物理隔离的两套路由 + 两套能力集**，权限边界由代码保证，不靠 prompt 约束。

## 2. 用户故事

- 作为**内部员工**，我用飞书扫码登录，能问内部知识、查客户、生成方案
- 作为**外部访客**，我从公众号进来，只能问公开产品知识，看不到任何客户/合同/内部文档
- 作为**管理员**，我能看审计日志，发现越权访问立即告警

## 3. 功能规格

### 3.1 登录 — 多渠道 IdP（E1.1 + E1.2）★

**决策（2026-06-22）：飞书 + 企业微信都接** → 做可插拔 OAuth 渠道抽象（符合 D2），不写死单一厂家。

```
IdP 渠道抽象：
  interface IdpChannel {
    name: 'feishu' | 'wecom'
    getAuthUrl(state): string
    exchangeCode(code): Promise<{ openId, email, name, deptId }>
  }
  实现：FeishuChannel / WecomChannel
  注册表：{ feishu: FeishuChannel, wecom: WecomChannel }

登录流程（渠道无关）：
  前端 → GET /api/internal/auth/:channel/login → 跳对应授权页
  回调 → GET /api/internal/auth/:channel/callback?code=&state=
  后端 → channel.exchangeCode(code) → 拿用户信息
       → upsert User（按 email 或 channel+openId 关联）
       → 签发 PAS JWT（含 uid/role/is_external）
  前端存 JWT → 后续 Authorization: Bearer
```

- 落地厂家时只改 `clients/idp/` 一个目录，业务层不动
- JWT payload：`{ uid, role, is_external, dept_id, exp }`
- 守卫：`@UseGuards(JwtAuthGuard)` + `@CurrentUser()` 注入用户
- User 表加 `auth_channel` + `auth_open_id`（同一人可能两渠道，按 email 归一）
- 外部渠道（公众号）：独立 token 机制，`is_external=true`，不走 IdP

### 3.2 内外路由树隔离（E1.3）★ 核心

```
apps/api/src/
 ├ internal/          # 只挂内部 module，JwtAuthGuard + 拒 is_external
 │   ├ qa/            # 内部问答
 │   ├ proposal/      # 方案
 │   ├ customer/      # 客户
 │   └ internal.module.ts
 ├ public/            # 只挂外部 module
 │   ├ qa-public/     # 公开问答（仅公开知识库）
 │   └ public.module.ts
 └ shared/            # 中性（health/feedback）
```

- `internal.module.ts` 全局守卫：`if (user.is_external) throw ForbiddenException`
- 物理隔离：外部用户即使拿到内部 URL，路由守卫直接 403
- **能力集隔离**：internal module 注入完整 clients（ragflow 全库 + crm）；public module 只注入 ragflow 公开库 client，无 crm

### 3.3 RBAC 角色（E1.4）

| 角色 | 权限 |
|------|------|
| `admin` | 全部 + 审计 + 用户管理 |
| `presales` 售前 | 问答 + 方案 + 客户 + 商机 |
| `aftersales` 售后 | 问答 + 工单（v2） |
| `implementation` 实施 | 问答 + 客户只读 |
| `external` 外部 | 仅公开问答 |

- 装饰器：`@Roles('presales','admin')` + `RolesGuard`

### 3.4 文档级 ACL（E1.5）★ 内容级权限

```
KbDocument.acl_scope: 'public' | 'internal' | 'role:presales' | 'dept:xxx'

检索时（E2.4 调用）：
  1. 根据 user 计算可见 scope 集合
  2. 调 ragflowClient.retrieve 时传 filters（按 ragflow_doc_id 白名单
     或 RAGFlow metadata 过滤）
  3. 越权文档不进 LLM 上下文
```

> 依赖 RAGFlow 是否支持检索时 metadata 过滤——若不支持，退化为"检索后按 PAS 的 KbDocument.acl_scope 二次过滤 chunk"。实测后定（任务书 §4.1）。

### 3.5 脱敏拦截器（E1.6）

- 全局 `SanitizeInterceptor`：响应序列化前，按 `user.is_external/role` 掩码
- 字段标注：DTO 上用 `@Sensitive({ maskFor: ['external'] })` 装饰器
- 规则：手机号 `138****5678`、邮箱 `a***@x.com`、合同金额对外隐藏
- **脱敏在 API 层完成，不在前端**（前端隐藏可被绕过）

### 3.6 审计日志（E1.7）

- 拦截器记录：`user_id / action / resource / is_external / detail`
- 触发告警：`is_external=true` 命中 internal 资源 → 写 AuditLog + 通知管理员

## 4. 数据模型增量

```prisma
model User {
  id          String   @id @default(cuid())
  email       String   @unique
  name        String
  role        String   // admin|presales|aftersales|implementation|external
  isExternal  Boolean  @default(false)
  deptId      String?
  createdAt   DateTime @default(now())
}

model AuditLog {
  id          String   @id @default(cuid())
  userId      String
  action      String
  resource    String
  isExternal  Boolean
  detailJson  Json?
  createdAt   DateTime @default(now())
  @@index([userId, createdAt])
}
```

## 5. 验收标准（DoD）

- [ ] 员工飞书登录成功，拿到含 role 的 JWT
- [ ] 外部 token 打 `/api/internal/*` 任意接口 → 403
- [ ] 外部用户问答响应 JSON 中无 11 位手机号 / 完整邮箱 / 合同金额
- [ ] 无权角色检索不到 `acl_scope=internal` 文档
- [ ] 越权访问在 AuditLog 留痕并触发告警
- [ ] 单测覆盖：守卫 / 脱敏拦截器 / ACL 过滤逻辑

## 6. 不在本模块（后置）

- 多租户隔离（v2）
- ABAC 细粒度策略（v2，先 RBAC + 文档级 ACL）
- 角色模板批量授权（v2）

# E2 售前问答 — 详细 spec

> 配套 [PAS-任务清单-V1.0.md](../../PAS-任务清单-V1.0.md) ｜ 里程碑 M1
> 依赖：E0（clients）、E1（权限/ACL）

---

## 1. 模块定位

PAS 的第一条端到端价值线，也是验证"PAS + RAGFlow"模式成立的最小闭环。员工问 IP-Guard 产品问题，系统从 RAGFlow 知识库检索，LLM 基于检索内容生成带出处的答案，支持多轮上下文。

## 2. 用户故事

- 作为**售后**，我问"控制台加密策略怎么设置"，得到带 UI 步骤 + 出处的答案
- 作为**售前**，我连续追问"延申一下""那它和 WPS 怎么配合"，系统理解上下文
- 作为**任一员工**，答案里引用的截图能直接看到，不用翻原 PDF

## 3. 功能规格

### 3.1 RAGFlow client 对接（E2.1）

```typescript
// clients/ragflow.ts
interface RagflowClient {
  retrieve(p: {
    query: string; kbId: string; topK?: number;
    docIdWhitelist?: string[]   // E1.5 ACL 过滤
  }): Promise<Chunk[]>

  chat(p: {
    messages: Message[]; kbId: string;
    docIdWhitelist?: string[]
  }): AsyncIterable<{ delta: string; refs?: Ref[] }>
}

interface Chunk {
  content: string
  docId: string
  docName: string
  page?: number
  images?: string[]   // 截图 URL（多模态）
  score: number
}
```

> 真实字段以 RAGFlow API 实测响应为准（任务书 §4.1）。

### 3.2 问答 API — SSE 流式（E2.2）

```
POST /api/internal/qa
body: { query, sessionId?, history? }
→ SSE 流：
   data: {"type":"delta","content":"加密"}
   data: {"type":"delta","content":"策略"}
   data: {"type":"refs","refs":[{n:1,docName:"Web控制台说明.pdf",page:24}]}
   data: {"type":"done"}
```

**编排逻辑（两种模式二选一，实测后定）**：
- **模式 A（用 RAGFlow chat）**：直接转发 RAGFlow 的 chat 流，PAS 只做鉴权/ACL/反馈
- **模式 B（PAS 自编排）**：PAS 调 retrieve 拿 chunk → 自己拼 prompt → 调 LLM 流式生成
  - 优点：prompt 可控（UI 步骤优先等业务规则）、可注入 ACL
  - 倾向 B，因为方案生成等模块也要复用"检索→LLM"这套

### 3.3 多轮上下文（E2.2）

- `history` 取最近 N 轮（默认 5 轮用户消息，控成本）
- 短指代（"延申一下""那个"）：把 history 一并喂给检索 query 改写或直接给 LLM
- `sessionId` 关联多轮，存 `ConversationFeedback`

### 3.4 出处引用（E2.3）

- 答案结论后标 `[1][2]`，文末列来源（文档名 + 页码）
- 引用可点击 → 展示该 chunk 原文 / 截图
- Prompt 约束：每个结论必须能溯源到检索 chunk，禁止臆测

### 3.5 ACL 过滤检索（E2.4）★

```
1. 取 user 可见 acl_scope（E1.5）
2. 查 PAS KbDocument 表 → 得可见 ragflow_doc_id 白名单
3. retrieve 时传 docIdWhitelist
4. （若 RAGFlow 不支持白名单过滤 → 检索后按 docId 过滤 chunk）
```

### 3.6 问答前端（E2.5）

- 聊天界面（流式逐字显示）
- Markdown 渲染（含 `![](url)` 截图自动展示）
- 出处列表可展开
- 多轮对话历史

### 3.7 反馈收集（E2.6）

- 每条答案下 👍/👎
- 写 `ConversationFeedback{ query, answer, rating, sessionId, userId }`
- 为后续"知识缺口分析"（v2）积累数据

## 4. 关键 Prompt（存 `prompts/qa_answer.txt`）

要点（不在代码里硬写）：
- 答案必须基于检索内容，禁止臆测
- 步骤/路径/字段名来自原文
- 每个结论标 `[n]` 引用
- **UI 操作步骤优先**，配置键放"高级配置（可选）"
- 检索不足以回答 → "该问题暂未收录，请联系工单系统"
- 涉及"授权软件"问题 → 先说明 IP-Guard 自带默认授权软件库（WPS/Office 等）

## 5. 验收标准（DoD）

- [ ] 问"控制台加密策略" → 答 UI 步骤 + 出处（非只给 `decrypt_files` 配置键）
- [ ] 连续 3 轮，第 3 轮"延申一下"能正确接上文
- [ ] 答案截图能在前端直接显示
- [ ] 越权用户检索不到受限文档（E1.5 联动）
- [ ] 拒答：检索不足时给兜底话术，不强答
- [ ] 👍/👎 入库
- [ ] 端到端响应 ≤ 30s

## 6. 不在本模块（后置）

- 知识缺口 admin 看板（v2）
- 用量/token 监控（v2）
- 公开问答 public 版（E1 路由就绪后单独做）

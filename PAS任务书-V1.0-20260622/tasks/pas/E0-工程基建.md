## 1. 模块定位

PAS 的代码地基。建好 monorepo 骨架、数据库、开发环境、配置体系、外部能力 client 封装、CI。完成后任何业务模块都能在统一约定下开工。

## 2. 功能规格

### 2.1 monorepo 骨架（E0.1）

```
pas/
 ├ apps/
 │   ├ api/          # NestJS 后端
 │   └ web/          # Next.js 15 前端
 ├ packages/
 │   └ shared/       # 共享 DTO / 类型 / 枚举（前后端复用）
 ├ docker-compose.dev.yml
 ├ pnpm-workspace.yaml
 └ .env.example
```

- 包管理：pnpm workspace
- `pnpm dev` 并发起前后端
- `packages/shared` 导出 `Customer`/`Proposal`/`User` 等类型，前后端共享（E9 类型安全）

### 2.2 Postgres + Prisma（E0.2）

- `apps/api/prisma/schema.prisma` 含任务书 §5 全部核心表
- 首个 migration 建表
- 各模块 spec 的"数据模型增量"在此合并

### 2.3 开发环境 compose（E0.3）

```yaml
# docker-compose.dev.yml
services:
  postgres:  # 业务库
  redis:     # 队列 + 缓存
  minio:     # 对象存储
```

- 端口避开已占用（RAGFlow/历史容器）
- 健康检查 + 数据卷持久化

### 2.4 配置体系（E0.4）

- `@nestjs/config` + zod schema 校验
- 缺必填 env → 启动即 fail fast
- `.env.example` 列全所有键（无默认值的标注必填）

### 2.5 clients 封装骨架（E0.5）★

```
apps/api/src/clients/
 ├ ragflow.ts   # interface + RagflowClientImpl + RagflowClientMock
 ├ llm.ts       # interface + 真实 + mock
 ├ crm.ts       # interface + ExternalCrmClient(第三方) + PasCrmClient(阶段二) + Mock
 └ index.ts
```

- 业务层只依赖 interface（DI 注入）
- **RAGFlow 用 mock 起步**，真实实现等实测后填——E2/E3/E4 不被阻塞
- LLM client 先接现有云端 API
- **crm client**：阶段一 `ExternalCrmClient` 接现有第三方 CRM（只读为主），由 `CRM_PROVIDER=external|pas` 选实现（E4 详述，厂家无关）

### 2.6 CI（E0.6）

- lint（eslint）+ typecheck（tsc）+ test（jest/vitest）
- PR 触发，全绿才可合

## 3. 验收标准（DoD）

- [ ] `pnpm dev` 前后端同起，浏览器访问到首页
- [ ] `docker-compose.dev.yml` 起 postgres/redis/minio，健康检查过
- [ ] `prisma migrate dev` 建出全部核心表
- [ ] 缺必填 env 时启动报错（非静默用默认值）
- [ ] mock RAGFlow client 可被业务层注入并跑通一个假问答
- [ ] CI 流水线跑通

## 4. 不在本模块

- 生产部署 compose（E5.2）
- 可观测性 / 监控（v2）

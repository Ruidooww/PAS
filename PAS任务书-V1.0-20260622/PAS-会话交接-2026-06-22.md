## 6. 协作流程计划（未落地，待定 2 项）

**模式**：Claude（我）当 orchestrator（架构/拆任务/写Issue/审PR/本地集成测试），Codex/Cursor 当 executor（实现→PR）。GitHub Issue/PR 当协作底座。

**关键事实**：
- Claude + Codex + Cursor 都在**同一台本机**，`gh` CLI 已 auth → 我能直接 `gh issue create` / `gh pr diff`
- 本地 agent **能连本地 RAGFlow/Postgres/CRM 做集成测试**（云沙箱不行）
- 自动化要"Issue→本地Codex"需 **self-hosted GitHub Actions runner**（云端 Actions 连不到 localhost）
- 不变：每个 PR 仍要人审；自动触发≠自动合并

**建议顺序**：① RAGFlow 实测过闸 → ② 建 PAS repo + E0 骨架 → ③ 手动跑通一次 Issue→Codex→PR→审 → ④ 验证后再上 self-hosted runner 自动化。**别在没跑通的流程上先配自动化。**

**待你定 2 项**：
1. PAS repo：私有/公开 + 名字（建议 `pas`）
2. CI executor：Codex（本地）还是 Claude Code Action（全本地+能本地测 → 倾向 Codex；Cursor 不进CI）
